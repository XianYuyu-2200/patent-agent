# 软件/AI 专利领域检查清单

本清单只补充软件、互联网、人工智能、数据处理和控制方法案件的领域判断。所有技术事实仍须满足调用方核心 Skill 的来源、状态、审批和输出合同。

## 1. 技术方案门禁

- [ ] 用可核验事实分别表述技术问题、技术手段和技术效果，不把经营目的、商业价值、用户偏好或管理目标改写成技术问题。
- [ ] 当方案只有会员、价格、营销、排序、推荐、结算、权限或其他业务规则，而缺少解决技术问题的技术手段和效果证据时，停止正式保护策略和起草，返回发明挖掘并生成补证问题。
- [ ] 当方案只是数学公式、模型概念、计算规则或其他抽象算法，且未与具体数据处理、计算机系统、硬件环境或被控制对象形成受支持的技术关联时，停止后续起草。
- [ ] “AI”“智能”“自动化”“大模型”等名称不证明存在模型、训练、推理或技术贡献；仅保存实际披露且有来源锚点的实现。

## 2. 数据输入、输出与处理流程

- [ ] 明确每项数据输入的定义、来源系统或对象、字段/格式、采集方式、时序、状态和来源锚点。
- [ ] 明确每项数据输出的内容、格式、接收组件或对象、用途、状态变化和来源锚点。
- [ ] 建立端到端数据处理链，逐项记录采集、预处理、传输、存储、计算、判断、反馈和异常处理步骤及其先后、条件和数据依赖。
- [ ] 对每个数据处理步骤记录执行主体、输入、处理动作、输出和与下一步骤的接口；不得用“处理模块”掩盖缺失算法或流程。
- [ ] 区分必要步骤、已披露可选步骤和无来源设想；把缺失参数、分支、异常路径、接口和数据对象转为待确认问题。

## 3. 模型训练与推理

- [ ] 仅在有证据证明使用机器学习模型时检查训练：训练任务、样本来源、特征/标签、清洗与划分、模型结构、损失函数、参数更新、评价指标和版本。
- [ ] 仅在有证据证明使用模型时检查推理：输入张量或特征、模型加载、推理步骤、输出含义、阈值/后处理、部署、更新和回滚。
- [ ] 记录训练与推理的关系、离线与在线边界、模型输出如何进入后续技术步骤，以及每一关系的来源锚点。
- [ ] 实际方案不使用机器学习时，将训练和推理标记为不适用，不得为符合“AI”名称而补造模型内容。

## 4. 执行环境与技术效果

- [ ] 明确软件组件、进程、服务、接口、处理器、存储器、网络节点或其他硬件环境；区分与技术贡献相关的环境和通用载体。
- [ ] 若方案控制设备、机器人、车辆、生产线、仓储系统或其他被控制对象，记录控制指令、目标对象、状态反馈和闭环关系；没有真实对象时不得虚构。
- [ ] 明确数据流和调用链如何作用于硬件环境或被控制对象，并检查该关联是否为解决技术问题所必需。
- [ ] 每项技术效果应有可追溯的结构或步骤因果链、比较基准、指标、测试条件和来源；不得把商业效果、用户体验口号或推测性优势当作已证实效果。

## 5. 权利要求客体对应

- [ ] 仅在共同技术构思和充分公开得到支持时布局方法、装置、设备和存储介质；四类客体不是默认必写清单。
- [ ] 方法权利要求按实际执行顺序表达必要数据、处理步骤、技术关联和输出，不只陈述目标或业务结果。
- [ ] 装置权利要求的模块或单元逐项对应方法步骤及其功能关系，不用无内容的模块名称扩大范围。
- [ ] 设备权利要求明确处理器、存储器及其执行的受支持程序方案；通用设备载体不能弥补基础方法缺乏技术性或公开不足。
- [ ] 存储介质权利要求只承载已经得到支持的程序方案，并与方法步骤保持一致。
- [ ] 核对方法、装置、设备和存储介质之间的术语、输入输出、步骤/模块、执行主体、依赖关系和必要特征逐项对应。

## 6. 阶段门禁

- [ ] 技术问题、必要数据处理、执行上下文或技术效果缺失，导致技术方案无法成立时，停在发明挖掘或相应上游阶段。
- [ ] 训练、推理、硬件环境或被控制对象仅是无来源包装时，删除其正式候选资格并生成追问；不得用行业常规补齐。
- [ ] 局部信息缺失但可与已批准权利要求支持分离时，按调用方核心 Skill 的合同记录缺口，不得擅自改变其 ready/blocked 判定。
- [ ] 领域检查不得放宽 `confirmed`、`source-backed`、审批、版本、新鲜度、输出数量或禁止编造规则。
