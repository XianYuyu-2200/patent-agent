# 中国专利撰写 Agent 设计

## 1. 目标

构建一个供专利服务人员内部使用的专业生产 Agent。客户可以提交技术交底书，也可以提交微信聊天、语音转写、图片、产品资料、代码等零散材料。Agent 协助完成技术挖掘、现有技术检索、保护策略设计、权利要求书和说明书撰写、质量审查及 DOCX 交付。

首期范围：

- 法域：中国；
- 申请类型：发明专利、实用新型；
- 技术领域：机械/设备/结构/电子硬件，以及软件/互联网/人工智能/控制方法；
- 输出目标：接近可提交的申请文件版本，但必须经过人工最终审核；
- 使用方式：Agent 是服务提供者的内部专业助手，不面向客户无人值守地自动提交申请。

## 2. 设计原则

1. 采用一个案件编排 Agent 加多个专业 Skills，不使用单一巨型提示词。
2. 技术事实、法律分析和撰写文本分层保存，避免推断被误写成客户事实。
3. 每个关键结论都应能追溯到客户材料或对比文献。
4. 工作流允许回退，不要求案件只能线性向前推进。
5. 权利要求发生实质修改后，必须重新执行说明书支持、引用关系和术语一致性检查。
6. Agent 可以生成追问、分析和建议，但不得补造客户未提供或未确认的技术事实。
7. 技术方案、权利要求和最终交付设置人工确认门禁。

## 3. 总体架构

系统由四层组成：

### 3.1 案件编排 Agent

案件编排 Agent 负责：

- 建立案件并维护当前阶段；
- 选择和调用专业 Skill；
- 管理术语表、技术特征编号和文档版本；
- 检查阶段输入是否满足要求；
- 管理人工审批与回退；
- 记录分析结论、材料来源和修改原因；
- 在上游产物变化后触发受影响阶段的回归检查。

编排 Agent 不应绕过专业 Skill 直接一次性生成全套申请文件。

### 3.2 核心 Skills

首期建立九个核心 Skills：

| Skill | 职责 | 主要输出 |
| --- | --- | --- |
| `cn-patent-case-intake` | 整理聊天、语音转写、图片、文档、代码和已有交底材料 | 案件档案、材料索引、缺失和冲突信息 |
| `patent-invention-mining` | 访谈发明人并识别技术问题、技术手段、效果和实施变体 | 发明点清单、技术特征树、追问清单 |
| `patent-prior-art-search` | 生成关键词、分类号和组合检索式，保存检索过程 | 检索日志、对比文件、证据锚点 |
| `patentability-analysis` | 分析新颖性、创造性、申请价值和风险 | 特征对比矩阵、可专利性意见 |
| `cn-claim-strategy` | 设计保护主题、独立权利要求和从属层级 | 保护布局、核心和兜底特征 |
| `cn-claim-drafting` | 起草和迭代中国专利权利要求书 | 权利要求书、特征来源映射 |
| `cn-specification-drafting` | 起草说明书、实施方式、摘要和附图说明 | 完整申请文件草稿 |
| `cn-patent-quality-review` | 检查支持性、清楚性、一致性、单一性和形式问题 | 分级问题清单、支持矩阵、修改建议 |
| `patent-document-export` | 套用模板并生成交付文件 | DOCX 文件包、客户确认稿、版本记录 |

### 3.3 领域专业包

领域专业包为核心 Skills 提供条件化规则、模板和检查项，不复制通用工作流。

`mechanical-hardware-patent` 包含：

- 部件、连接、位置、运动和配合关系建模；
- 结构方案与方法方案的保护选择；
- 附图规划、部件标号和文字一致性检查；
- 实用新型适用性及结构特征检查。

`software-ai-patent` 包含：

- 技术问题、技术手段和技术效果识别；
- 数据采集、预处理、训练、推理和控制流程建模；
- 方法、装置、设备和存储介质等保护客体布局；
- 业务规则、抽象算法和可专利技术方案的区分；
- 软件流程与硬件环境、数据对象及控制对象的关联检查。

### 3.4 共享基础资源

- 中国专利法律规则、审查指南和内部撰写规范；
- 机械/硬件与软件/AI 的交底和访谈模板；
- 权利要求、说明书、摘要和质量审查模板；
- DOCX 交付模板；
- 统一案件数据结构；
- 术语表、技术特征标识和来源锚点规范；
- 可复用但经过脱敏的案例模式和质量问题库。

详细规则应放入 Skills 的 `references/`，稳定且重复的格式转换和检查逻辑应放入 `scripts/`，交付模板应放入 `assets/`。`SKILL.md` 只保留核心流程、调用条件和质量门禁。

## 4. 八阶段工作流

### 阶段 1：客户受理与材料归档

接收各类客户材料，提取案件主体、发明人、申请类型、技术领域、公开时间和材料来源。输出材料清单、缺失信息、冲突信息及保密风险提示。

进入下一阶段前，必须确认申请主体、发明人信息、已公开情况和基本保密风险。

### 阶段 2：发明挖掘与访谈

从材料和访谈中识别现有问题、技术目标、核心技术手段、技术效果、替代实施方式和商业重点。机械案件形成部件及关系模型；软件案件形成数据、步骤、模块和运行环境模型。

人工门禁 A：服务提供者与发明人确认技术事实、发明点和关键实施方式。

### 阶段 3：现有技术检索与对比

围绕核心特征和特征组合设计关键词、同义词、申请人、发明人、IPC/CPC 分类号及组合检索式。记录检索数据库、检索式、日期和筛选过程，并为关键结论保存文献段落、权利要求或附图位置。

### 阶段 4：保护策略与申请布局

基于技术事实和检索结果，确定保护客体、核心发明构思、必要技术特征、独立权利要求数量、从属层级、兜底方案，以及发明和实用新型的组合申请策略。

检索结果不足、核心方案已公开或技术事实不完整时，返回技术挖掘或检索阶段。

### 阶段 5：权利要求书起草

先起草独立权利要求，再根据商业价值、创造性贡献、实施变体和规避空间构建从属权利要求。每个技术特征记录其客户材料来源、说明书支持位置和策略作用。

人工门禁 B：服务提供者确认保护范围、必要技术特征、权利要求层级和潜在规避空间。

### 阶段 6：说明书、摘要与附图

以已确认的权利要求和技术事实为约束，完成技术领域、背景技术、发明内容、附图说明、具体实施方式和摘要。规划机械结构图、流程图、模块图等附图，并维护标号表。

实施例只能基于客户材料或经确认的合理展开。尚未确认的内容必须保持显式标记，不得作为确定事实写入最终稿。

### 阶段 7：独立质量审查

以审查员和竞争对手视角检查：

- 权利要求的清楚性、引用关系和保护范围；
- 说明书对权利要求的支持及充分公开；
- 权利要求、说明书、摘要、附图及标号的一致性；
- 新颖性、创造性、单一性和保护客体风险；
- 机械结构或软件流程中的逻辑缺口；
- 容易规避、无必要限缩或缺少兜底层级的问题。

质量审查应使用相对独立的上下文，避免继续为原稿辩护。

### 阶段 8：交付、版本与知识沉淀

生成规范 DOCX 文件、客户确认清单、内部审核记录和版本差异说明。保留全过程材料来源、人工审批、修改理由和质量问题关闭记录。

人工门禁 C：服务提供者完成最终专业审核后才能对外交付或进入申报流程。

## 5. 案件数据结构

每个案件包含六类核心数据：

1. `source-materials`：客户原始材料，只读保存；
2. `technical-facts`：提取并确认的技术事实；
3. `feature-tree`：技术特征、关系、步骤和可选变体；
4. `prior-art`：检索式、对比文件、证据位置和特征矩阵；
5. `drafts`：保护策略、权利要求书、说明书和摘要版本；
6. `review-log`：问题、修改原因、影响范围和审批记录。

关键内容必须带以下来源状态之一：

- `confirmed`：已经由客户或服务提供者确认；
- `source-backed`：可以定位到原始材料；
- `inferred`：由 Agent 推断但尚未确认；
- `missing`：缺少必要信息；
- `conflicted`：不同材料存在冲突。

`inferred`、`missing` 和 `conflicted` 内容不得直接作为确定技术事实进入最终申请文件。

## 6. 阶段接口

Skills 不直接依赖完整聊天上下文，而通过结构化案件产物交接：

```text
案件信息
→ 技术事实库
→ 技术特征树
→ 对比文件及证据
→ 保护策略
→ 权利要求版本
→ 说明书版本
→ 质量问题与处理记录
```

每次 Skill 调用至少记录：

- 输入产物及版本；
- 使用的领域专业包；
- 输出产物及版本；
- 未解决问题；
- 来源锚点；
- 是否需要人工审批；
- 哪些下游产物因本次修改而失效。

## 7. 回退和错误处理

- 核心方案被现有技术覆盖：返回技术挖掘或保护策略阶段；
- 权利要求缺乏说明书支持：补充经确认的实施内容或缩小保护范围；
- 实施例需要新增技术事实：生成客户追问，不自动编造；
- 独立权利要求发生实质修改：重跑从属关系、说明书支持和术语一致性检查；
- 机械附图与文字不一致：冻结交付并重新核对部件和标号；
- 软件方案只有业务规则而缺少技术手段：返回技术挖掘阶段；
- 检索服务不可用：保留当前检索计划和未完成状态，不将不完整检索包装成确定结论；
- 文档解析置信度不足：保留原始页面或时间戳锚点，并要求人工核对。

## 8. 质量验收标准

交付前必须满足：

- 每项权利要求特征都有客户材料来源或说明书支持；
- 独立权利要求包含解决技术问题所需的必要技术特征；
- 权利要求引用关系正确；
- 权利要求、说明书、摘要和附图术语一致；
- 附图标号无遗漏、重复和指代冲突；
- 背景技术没有误写为本申请方案；
- 实施方式能够支撑要求保护的范围；
- 新颖性和创造性判断具有可追溯证据；
- 所有待确认、缺失和冲突标记均已关闭或明确接受风险；
- 高风险质量问题已经关闭；
- 最终版本经过服务提供者人工审核。

## 9. 测试策略

### 9.1 Skill 单元测试

为每个 Skill 准备机械/硬件和软件/AI 的典型样例，验证输入字段、输出结构、缺失信息处理和禁止编造规则。

### 9.2 契约测试

验证上游 Skill 的输出能够被下游 Skill 消费，并检查术语编号、来源锚点和版本字段没有丢失。

### 9.3 黄金案件回归测试

选择已经完成且允许脱敏使用的真实案件，建立人工认可的技术特征树、保护策略、权利要求结构和质量问题清单。每次更新 Skill 后比较关键差异，不要求文字逐字一致。

### 9.4 对抗测试

覆盖以下情形：材料相互矛盾、客户遗漏关键参数、已有公开、机械关系描述含糊、软件方案只有商业规则、权利要求范围过宽、实施方式不足、附图标号错误及检索证据缺失。

### 9.5 人工验收指标

首期主要评估：

- 需要人工纠正的技术事实数量；
- 权利要求重大重写次数；
- 支持性和一致性问题数量；
- 从材料接收到可审核初稿的时间；
- 客户补充沟通轮数；
- 最终交付前仍未关闭的高风险问题数量。

## 10. 首期不包含

- 自动向国家知识产权局提交申请；
- 无人工审核的客户自助交付；
- 美国、欧洲和 PCT 多法域规则；
- 化学、材料和生物医药专业撰写；
- 审查意见答复自动化；
- 费用、合同、开票和客户关系管理系统。

系统应为后续增加审查意见答复、案件历史复用和多法域扩展保留版本及证据接口，但首期不实现这些业务能力。
